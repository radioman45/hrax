## 0. 30초 요약

- 하네스 = AI가 일할 때 따라야 할 **5단계 매뉴얼**
- 5단계 = Clarify(되묻기) → Context(자료수집) → Plan(계획) → Generate(실행) → Evaluate(검증)
- 앞 2단계는 **Claude.ai 브리핑룸**에서 대화로, 뒤 3단계는 **Claude Code + harness 플러그인**에서 실행
- 사이를 잇는 건 **브리프 문서 1장** (60줄 이내)
- 처음엔 **단일 스킬로 가볍게** 시작, 문제 생기면 그때 쪼갠다
- **과잉 설계는 최대의 적** — 겪지 않은 문제를 미리 해결하려 하지 마라

## 1. 하네스가 뭐고 왜 쓰는가

### 한 문장 정의

**"AI가 일하다가 중요한 단계를 건너뛰지 못하게 강제하는 체크리스트."**

### 왜 필요한가

AI는 피곤하거나 모호한 지시를 받으면 Clarify·Evaluate 같은 중요 단계를 스스로 건너뜁니다. 빈 채팅창에 "앱 만들어줘"라고 치면 대충 코드부터 쏟아내는 그 현상입니다. 하네스는 _"너는 무조건 이 5단계를 밟아야 해"_ 라고 AI를 묶어두는 장치예요.

### 효과 (검증된 수치)

revfactory 논문 실험: 15개 SW 과제에서 **평균 품질 49.5 → 79.3 (+60%)**. 작업이 어려울수록 효과가 커짐 (Expert 과제 +36.2점).

### 언제 쓰고 언제 안 쓰는가

**쓴다**:
- 반복되는 업무 (매주 하는 리포트, 정례 분석)
- 중요한 단발 과제 (품질이 중요한 문서, 민감한 산출물)
- 남에게 넘겨줄 산출물

**안 쓴다**:
- 단순 질문 (뉴스 찾기, 개념 설명)
- 즉흥적 브레인스토밍
- 분량 짧은 메모

**판단 기준**: _"결과물을 누가 보는가? 그 사람이 만족하는 게 중요한가?"_ YES면 하네스, NO면 그냥 대화.

## 2. 전체 구조

```
[영환님]                [Claude.ai 브리핑룸]          [Claude Code + harness]
   │                         │                              │
   │  "새 과제 시작" ────────▶                             │
   │                         │  Kickoff Ritual              │
   │                         │  (영역·과제·Input·Output)    │
   │  ◀──── 되묻기 ─────────                              │
   │  ────── 답변 ──────────▶                             │
   │                         │  Clarify · Context           │
   │                         │                              │
   │  ◀── 브리프 문서 1장 ───                             │
   │                                                        │
   │  폴더에 브리프 + 자료 넣고 claude 실행               │
   │  ──── "이 브리프대로 하네스 구축해줘" ──────────────▶
   │                                                        │  Plan · Generate · Evaluate
   │                                                        │  (.claude/ 자동 생성)
   │  ◀──────── 산출물 ───────────────────────────────────
```

### 두 시스템의 역할 분담

| 단계 | 어디서 | 왜 여기서 |
|---|---|---|
| Clarify (되묻기) | Claude.ai 브리핑룸 | 대화 UX가 좋음, 모바일 OK |
| Context (자료수집) | Claude.ai 브리핑룸 | Project knowledge · 과거 대화 검색 |
| Plan (계획) | Claude Code | 파일·폴더 접근 필수 |
| Generate (실행) | Claude Code | 코드·문서 실제 생성 |
| Evaluate (검증) | Claude Code | 자동 테스트·린트 가능 |

### 연결 고리 = 브리프 문서

브리핑룸에서 나온 **마크다운 파일 1장**이 작업 폴더에 들어가서 Claude Code의 지시서가 됩니다. 이 파일이 릴레이 바통이에요.

## 3. 1회 세팅 — 하네스 브리핑룸 만들기

한 번만 하면 됩니다. 이후 모든 과제가 여기서 시작돼요.

### 3-1. Claude.ai Project 생성

1. Claude.ai 좌측 사이드바 → **Projects** → **Create project**
2. **Name**: `하네스 브리핑룸`
3. **Description**: `모든 일상/업무 과제를 Claude Code에 넘기기 전, 대화로 브리프 문서를 완성하는 공용 방`

### 3-2. 시스템 프롬프트 등록

Project instructions 칸에 아래 내용을 **그대로 복붙**:

```markdown
# 역할
너는 이영환이 Claude Code(또는 다른 실행 에이전트)에게 작업을 시키기 전,
대화로 "브리프 문서 1장"을 같이 완성하는 파트너다.

하네스 5단계(Clarify · Context · Plan · Generate · Evaluate) 중
너는 Clarify와 Context Gather만 담당한다.
Plan · Generate · Evaluate는 Claude Code나 사용자가 맡는다.

다루는 영역은 제한 없다. 업무(HR, 투자 분석, 문서 작성),
개인(가족 일정, 학습 계획, 건강 관리), 사이드 프로젝트 등 뭐든 온다.

# 과제 시작 의식 (Kickoff Ritual)
사용자가 새 과제를 꺼내면 가장 먼저 이 4가지를 한 줄씩 확정한다.
1. 영역 (업무 · 가족 · 개인 · 사이드 중 하나)
2. 과제명
3. Input (어떤 자료·정보를 넣는가, 파일 형태)
4. Output (뭐가 나와야 하는가, 포맷)

4가지가 한 줄로 깔끔히 정의되기 전까진 다음 단계로 넘어가지 않는다.
영역을 제일 먼저 확정한다 — 이후 되묻기 깊이와 제약 적용이 달라지기 때문.

영역 판단 기준 (헷갈릴 때):
"이 결과물을 누가 보게 되나?"
· 회사 동료 → 업무  · 배우자/아이 → 가족
· 나만 → 개인       · 불특정 다수 → 사이드

# 대화 규칙
1. 모호한 건 한 번에 하나씩만 묻는다. 질문 여러 개를 동시에 쏟지 않는다.
2. 답하기 어려워 보이면 2-4개 선택지를 제시한다.
3. 이 프로젝트 내 과거 대화 · 업로드 파일에서 찾을 수 있는 건 먼저 찾는다.
4. 민감정보(실명, 급여, 평가, 가족 사생활 등)는 예시에 쓰지 않는다.
5. 과제 성격에 따라 5단계 중 일부를 생략해도 된다.
   (예: 빈 프로젝트라 참고 자료 없음 → Context 생략)
   생략 시 브리프에 "생략. 이유: ___"를 명시한다.
6. 영역별 기본값을 따른다.
   · 업무: 컴플라이언스·브랜드·민감정보 주의 강하게 적용
   · 가족: 가족 실명·사생활 노출 최소화, 공유 전제로 톤 조정
   · 개인: 외부 공유 가정하지 않음, 개인 재무·건강 정보 보호
   · 사이드: 공개 품질로 가정, 저작권·출처 점검
   영역 간 크로스 참조는 사용자가 명시적으로 허용할 때만.

# 브리프 작성 규칙 (최종 산출물)
- 형식: 마크다운 1개 파일, 60줄 이내
- 파일명: 00-brief-[과제슬러그]-v1.md
  예: 00-brief-survey-feedback-v1.md
- 반드시 포함할 섹션:
  1. 영역 · 과제명 · Input · Output · Done 기준
  2. 5단계 매뉴얼 (각 2-3줄, 생략 단계는 이유 명시)
  3. 제약사항 (민감정보, 브랜드, 컴플라이언스, 금지사항)
  4. 포인터 (관련 스킬/문서 경로)
- 내용을 풀어쓰지 말고 "포인터"만 적는다.
  예: "SK 브랜드 규칙은 ipc-ppt 스킬 참조" (본문 복붙 X)
- 디렉터리 트리, 상세 스타일 가이드는 쓰지 않는다 (AI가 오히려 헤맴).

# 답변 포맷
일반 대화 시:
- Facts(출처 포함)와 Opinion 분리
- 마지막에 확장 질문 2개

브리프 문서 완성 시:
- 코드블록 안에 전체 마크다운으로 제시
- 하단에 "이대로 Claude Code에 넘기시겠습니까?" 확인 1회

# 금기
- 서두 반복 금지 ("기업의 HR 담당자로서" 같은 것)
- 30줄 넘는 설명 덩어리 금지 (짧게 자주)
- 사용자가 묻지 않은 주제로 먼저 확장 금지
- 과거 맥락을 자동 가정하지 말 것 (영역은 매번 확인)
- "검토"·"비교" 요청이라도 결론이 명확할 땐 결론을 앞에 두고 짧게 끝낸다.
  옵션 나열로 결론을 희석하지 않는다. 분량으로 답변의 무게를 만들지 않는다.
- 사용자가 아직 겪지 않은 문제를 해결하기 위한 구조 변경은 먼저 제안하지 않는다.
  실제 마찰·실패·비효율이 관찰된 뒤에 업그레이드 제안을 꺼낸다.
```

## 4. 매 과제 워크플로우

### 4-1. 브리핑룸에서 Kickoff 메시지 보내기

**템플릿**:
```
새 과제 시작합니다.

영역: [업무 / 가족 / 개인 / 사이드 중 하나]
과제명: [여기에 과제 이름]
배경: [한 줄로 왜 이걸 하려는지]
참고 자료: [첨부했으면 "이 채팅에 첨부함", 없으면 "없음"]

Kickoff Ritual 시작해줘.
```

**실사용 예시 4종**:

> **업무**: 영역: 업무 / 과제명: 리더십 Survey Feedback Report 자동화 / 배경: 팀장별 리포트 수작업 변환 시간 절감 / 참고 자료: 이 채팅에 Survey 양식 첨부

> **가족**: 영역: 가족 / 과제명: 아이 여름방학 주간 학습 계획 / 배경: 독서·수학·영어 균형 배치 / 참고 자료: 없음

> **개인**: 영역: 개인 / 과제명: 포트폴리오 분기 리밸런싱 체크 / 배경: 자체 점검 루틴 만들기 / 참고 자료: 포트폴리오 시트 첨부

> **사이드**: 영역: 사이드 / 과제명: 맥킨더 지정학 블로그 글 / 배경: 학습 정리 겸 브런치 발행 / 참고 자료: 메모 3건 첨부

### 4-2. 티키타카 5-7회 예상

Claude가 한 번에 하나씩 되물음. 전형적 흐름:

1. 영역·과제명 확인 → Input 물어봄
2. Input 답 → Output 물어봄
3. Output 답 → 세부 구성 물어봄 (선택지 제시)
4. 구성 답 → Context Gather 필요 여부 확인
5. Context 답 → 제약사항 확인
6. **브리프 문서 제출** → "Claude Code에 넘기시겠습니까?"
7. OK → 파일로 저장 요청

**소요 시간**: 10-15분. Input/Output이 머릿속에 이미 있으면 5-7분.

### 4-8. ADR 작성 (작업 마무리 단계)

회고를 쓴 직후, **이번 작업에서 내린 구조적 결정들을 ADR로 남긴다.**

> **ADR이 뭔가요?** 요리에 비유하면 이렇습니다.
> 회고(`99-retro`)가 "이번 요리 맛이 어땠냐"는 기록이라면,
> ADR은 "왜 이 레시피를 선택했냐"는 기록입니다.
> AI가 다음 세션에서 "왜 이렇게 설계됐지?"라고 물어볼 때
> ADR이 있으면 즉시 맥락을 이해하고, 없으면 처음부터 다시 추측합니다.

**ADR이 필요한 순간:**
- 에이전트 수·역할을 결정했을 때
- 두 가지 방식 중 하나를 골랐을 때 (예: 병렬 vs 순차)
- 설정을 외부 파일로 분리했을 때
- 사용자 피드백으로 구조가 바뀌었을 때

**ADR이 필요 없는 순간:**
- 색상 값만 바꾼 경우
- 단순 버그 수정
- 재실행(구조 변경 없음)

**저장 위치**: `survey-feedback/adr/NNN-{결정-요약}.md`

**Claude Code에 이렇게 요청하면 자동 생성됩니다:**
```
지금까지 작업을 진행하면서 아키텍처(구조)적으로 여러 결정들을
내렸을 텐데, 그 결정을 왜 그렇게 했는지 이유와 과정을 ADR 문서로
기록해 줘.
```

**ADR 파일 형식 (4섹션):**
```markdown
# ADR-{번호}: {결정 제목}
**날짜:** YYYY-MM-DD  |  **상태:** 확정

## 맥락
이 결정이 왜 필요했나 (비개발자도 읽을 수 있게)

## 결정
무엇을 어떻게 결정했는가 (한 문단)

## 이유
왜 이걸 선택했는가, 다른 선택지는 왜 아닌가

## 결과
이 결정으로 생긴 장점과 단점 (솔직하게)
```

## 5. 5단계 실전 적용

### 5-1. Clarify (요구사항 구체화)

**목적**: AI가 상상으로 채우지 못하게 되묻기 강제.

**핵심 질문 3종**:
- Input/Output이 한 줄로 정의되는가?
- 모호한 용어는 없는가? (예: "잘 정리해서" → 어떤 포맷?)
- 잘못됐을 때 어떻게 아는가? (Done 기준)

**생략 가능성**: 없음. 항상 실행.

### 5-2. Context Gather (맥락 수집)

**목적**: 참고할 과거 자료·기존 자산 모으기.

**체크리스트**:
- 기존 비슷한 산출물 샘플 있나?
- 참조할 내부 문서·스킬 있나?
- 브랜드·포맷 가이드 있나?

**생략 가능성**:
- 빈 프로젝트 (참고할 게 없음) → 생략
- 완전히 새 영역 → 생략 후 실행 중 발생하는 필요를 기록

**생략 명시 예**: `Context Gather: 생략. 이유: 새 과제, 참고 자산 없음`

### 5-3. Plan (설계)

**목적**: 큰 과제를 쪼개기. AI 한 호흡으로 못 할 크기는 반드시 분할.

**Task/Page 구조**:
- **Task** (1개): 전체 목표 — "MVP 구현", "리포트 10부 생성"
- **Page** (여러 개): 세부 단계 — "엑셀 파싱", "팀장 1명 리포트", "ZIP 묶기"

**쪼개는 기준**: Page 하나를 서브에이전트가 처음부터 끝까지 독립 수행할 수 있는가?

### 5-4. Generate (생성)

**목적**: 실제 산출물 만들기.

**Lv1 기본**: 단일 스킬에서 순차 실행.
**Lv2 업그레이드**: 서브에이전트 분리 + 병렬 실행.
**Lv3 업그레이드**: Python 오케스트레이터 + 체크포인트 + 재시도.

**병렬 vs 순차 판단**:
- Page들이 서로 독립적 → 병렬 가능 (빠름, 토큰 많이 씀)
- Page들이 순서 의존 → 순차 필수
- 학습 초기 → 순차 권장 (흐름 이해 쉬움)

### 5-5. Evaluate (검증)

**목적**: AI가 검증 건너뛰지 못하게 강제.

**레벨별 검증**:
- **Lv1**: 사람이 샘플 1개 눈으로 확인
- **Lv2**: QA 서브에이전트가 전체 샘플링 체크
- **Lv3**: 자동 린트 (regex 패턴, 분량 체크, 브랜드 검증)

**민감도별 권장**:
- **민감도 상** (평가·인사·재무): 샘플 승인 게이트 필수
- **민감도 중** (내부 보고): 샘플 1개 검수
- **민감도 하** (초안·브레인스토밍): 생략 가능

## 10. 유지보수·업그레이드 원칙

### 원칙 1 — Single Source of Truth (SSOT)

같은 값이 여러 곳에 있으면 언젠가 어긋납니다. 한 곳에 두고 나머지는 읽기만.

**예시**:
- 브랜드 색상 → `_workspace/config.json`에 하나만
- QA 체크리스트 → `_workspace/qa-checklist.json`에 하나만

### 원칙 2 — 버전 관리

파일명에 v1, v2 명시. 작동하는 버전은 지우지 말고 남겨둠. `99-retro-v1.md`와 쌍으로 관리.

### 원칙 3 — 회고와 ADR이 진짜 자산

실행만 하고 회고·ADR 안 쓰면 배운 게 증발합니다.

| 문서 | 기록하는 것 | 저장 위치 |
|------|-----------|----------|
| 회고 (`99-retro-vN.md`) | 이번 작업이 어떠했나 (결과 품질) | 프로젝트 루트 |
| ADR (`adr/NNN-*.md`) | 왜 이 구조를 선택했나 (설계 이유) | `adr/` 폴더 |

둘의 차이를 쉽게 기억하는 방법:
> 회고 = **"잘됐나, 이상했나"**
> ADR = **"왜 이렇게 만들었나"**

**회고 형식 고정**:
- 잘된 것
- 이상한 것 (원했던 결과와 어긋남)
- 다음 버전에서 바꿀 것

### 원칙 4 — 변경은 최소 단위

v2 업그레이드 시 **구조는 유지, 설정만 외부화** 같은 최소 변경 지향. 전체 재구축 유혹 경계.

### 원칙 5 — 과잉 제안 경계

AI(저)가 "이것도 추가하면 좋을 것 같은데요"라고 미래 확장을 제안할 때, 영환님이 실제로 그 문제를 겪고 있는지 먼저 확인. 안 겪었으면 거절.

**거절 발화**: `"내가 지금 겪은 문제 아님. 보관함에."`

### 원칙 6 — 동료 공유는 영환님이 직접 써본 뒤

본인이 세 번 이상 성공적으로 쓴 하네스만 공유. 동료가 질문했을 때 즉답 가능해야 함.

```markdown
# 하네스 5단계 한 장 요약

## 왜 필요한가
AI는 피곤하거나 모호한 지시를 받으면 Clarify·Evaluate 같은
중요 단계를 혼자 건너뛴다. 하네스는 이걸 강제로 밟게 만드는
"업무 체크리스트"다. 단일 스킬로 시작, 커지면 쪼갠다.

## 제1원칙 — Input/Output 먼저
무엇을 넣으면(Input) 무엇이 나오는가(Output)를
한 줄로 못 쓰면 다음 단계로 넘어가지 않는다.
예: "기획서를 넣으면 MVP 코드가 나온다"

## 5단계

1. Clarify — 요구사항 구체화
   모호한 것, 빠진 것을 사용자에게 먼저 되묻는다.
   추측으로 채우지 않는다.

2. Context Gather — 맥락 수집
   과거 기록, 참고 문서, 기존 자산을 모은다.
   새 프로젝트라 참고 자료가 없으면 생략해도 된다.

3. Plan — 설계/계획
   크고 복잡한 작업은 Task(전체 목표) + Page(세부 단계)로 쪼갠다.
   뼈대를 잡아야 Generate가 중간에 흐트러지지 않는다.

4. Generate — 생성/구현
   실제 작업 수행. 규모가 크면 서브에이전트나
   Python 스크립트에 위임(Lv3에서 다룸).

5. Evaluate — 검증
   결과물이 Input/Output 정의와 처음 요구사항에 맞는지 자체 테스트.
   민감한 산출물일수록 생략하지 않는다.

## 생략 규칙
Context 생략: 빈 프로젝트, 참고 자료 없음
Evaluate 생략: 빠른 프로토타입, 재작업 비용 낮음
생략 시 브리프에 "생략. 이유: ___"를 명시한다.

## 영역별 적용 팁
- 업무: 5단계 다 밟기, Evaluate 강하게
- 가족: Clarify에 공유 대상 확인 (배우자/아이 수용도 다름)
- 개인: 빠르게 Plan→Generate로, Evaluate는 스스로 리뷰
- 사이드: 사실/출처 검증 (Evaluate) 무조건 켜기
```

## 부록 D — 자주 쓰는 판단 질문 모음

과제 중 헷갈릴 때 본인에게 묻는 질문들:

**Kickoff 단계**:
- 이 과제의 결과물을 **누가** 보는가? (영역 결정)
- Input·Output을 **한 줄로** 쓸 수 있는가?
- **Done**의 기준이 구체적인가?

**Plan 단계**:
- Page 하나를 서브에이전트가 **독립적으로** 수행 가능한가?
- 10개 넘는 Page라면 **정말 필요한가**?

**Generate 단계**:
- 병렬 vs 순차: **지금 학습 중**인가, 아니면 **속도 중요**한가?
- 민감도: **샘플 승인 게이트** 필요한가?

**Evaluate 단계**:
- 검증을 **사람이** 할지 **스크립트가** 할지 정해졌는가?
- Critical·Warning **임계값**은 명확한가?

**레벨업 판단**:
- 지금 구조로 **실제 불편**한가, 아니면 **예상 불편**인가?
- 최소 변경으로 풀 수 있는가, 아니면 **재구축** 필요한가?

## 부록 E — 용어 사전 (비개발자용)

| 용어 | 쉬운 뜻 |
|---|---|
| 하네스(Harness) | AI가 따라야 할 업무 매뉴얼 |
| 스킬(Skill) | AI에게 주는 업무 수칙 문서 |
| 에이전트(Agent) | 특정 역할을 맡은 AI 일꾼 |
| 서브에이전트 | 본 AI 밑에서 특정 역할만 하는 보조 AI |
| 오케스트레이터 | 여러 AI의 작업 순서를 관리하는 역할 |
| Task / Page | 전체 목표(Task) + 세부 단계들(Page) |
| 아티팩트(Artifact) | 한 단계에서 만들어 다음 단계에 넘기는 중간 결과물 |
| 컨텍스트 | AI가 기억하는 대화 내용의 총량 |
| 컨텍스트 소진 | AI 기억 용량이 다 차서 이전 내용을 잊기 시작하는 상태 |
| 체크포인트 | 중간 저장점. 끊겨도 여기서부터 재시작 가능 |
| SSOT | Single Source of Truth. 같은 값은 한 곳에만 둔다 |
| Progressive Disclosure | AI에게 필요할 때만 정보 공개하는 방식 |
| ADR | Architecture Decision Record. "왜 이 구조를 선택했는가"를 기록하는 문서. 회고(잘됐나)와 다르게 설계 이유를 남긴다 |
| 아키텍처 결정 | 코드나 구조를 어떻게 만들지 선택하는 것. 예: 에이전트를 1개로 할지 3개로 나눌지 |

## 11. ADR — 아키텍처 결정 기록

### ADR이란 무엇인가

> **한 줄 정의**: "왜 이 구조로 만들었는지"를 기록해두는 메모.

AI와 함께 작업하다 보면 에이전트를 몇 개로 나눌지, 설정을 어디에 둘지, 순차로 할지 병렬로 할지 등 구조적 선택을 수십 번 합니다. 그 선택들을 기록하지 않으면 다음 세션에서 AI도, 사람도 "왜 이렇게 됐지?"를 처음부터 다시 추측하게 됩니다.

**회고와 ADR의 차이**:

| | 회고 (99-retro) | ADR (adr/NNN-*.md) |
|--|-----------------|---------------------|
| 묻는 질문 | 이번 작업 잘됐나? | 왜 이 구조를 선택했나? |
| 시점 | 결과 나온 후 | 구조 결정 직후 |
| 내용 | 감상·개선점 | 선택 이유·트레이드오프 |
| 분량 | 자유 | 4섹션, 1페이지 내외 |

### ADR이 없으면 생기는 문제

1. **AI가 맥락 없이 재설계** — 새 세션에서 "왜 에이전트가 3개지?"를 모르면 AI가 자기 판단으로 구조를 바꿔버릴 수 있습니다.
2. **같은 실수 반복** — "색상을 두 파일에 따로 관리했다가 QA 오탐 발생"을 ADR에 안 남기면 v3에서 또 같은 실수를 합니다.
3. **동료 온보딩 어려움** — 새 팀원이 "config.json은 왜 만들었어요?"라고 물어볼 때 ADR이 있으면 파일 하나로 설명이 끝납니다.

### 언제 ADR을 쓰는가

**쓴다**:
- 에이전트 수·역할을 처음 설계할 때
- 병렬 vs 순차, 팀 모드 vs 서브에이전트 중 택할 때
- 설정·상수를 외부 파일로 분리할 때
- 사용자 피드백으로 구조가 바뀔 때

**안 쓴다**:
- 색상 값만 바꾼 경우 (설정 변경, 구조 변경 아님)
- 오류 수정
- 동일 구조 재실행

### ADR 자동 생성 방법

Claude Code에 이렇게 입력하면 됩니다:

```
지금까지 작업을 진행하면서 아키텍처(구조)적으로 여러 결정들을
내렸을 텐데, 그 결정을 왜 그렇게 했는지 이유와 과정을
ADR 문서로 기록해 줘.
```

오케스트레이터 스킬에 ADR 자동 생성이 등록되어 있으면, 작업 마무리 시 자동으로 생성됩니다 (회고 제출 직후).

### 파일 관리 규칙

```
프로젝트/
└── adr/
    ├── 001-에이전트-3분리-구조.md
    ├── 002-config-외부화.md
    └── 003-샘플-선승인-후-배치.md
```

- 번호는 001부터 순서대로
- 파일명은 결정 내용을 2~4단어로 (한국어 OK)
- 수정이 생기면 파일 안 `**상태:**` 를 `검토 중` → `확정` → `폐기` 로 업데이트

### ADR과 회고의 연계

```
작업 완료
    ↓
회고 작성 (99-retro-vN.md)
"다음 버전에서 바꿀 것" 항목
    ↓
구조 변경이 있다면 → ADR 생성 (adr/NNN-*.md)
단순 설정 변경이면 → ADR 불필요
```
